# AI 探索功能演进史：问题、挑战与解法

> 本文记录「AI 探索生成接口测试用例」这条功能线的演进历程——遇到的问题、关键挑战、定位反复，以及每一步为什么这么解。
> 与 [DESIGN_DECISIONS.md](DESIGN_DECISIONS.md) 互补：那里记"最终决策的为什么、约束、反例"，这里记"我们是怎么一步步走到那些决策的"。
> 读它能避免重蹈定位反复，也能理解当前形态背后的取舍。

---

## 一句话定位（当前）

> 从**业务意图**出发（需求文档 / 主干链路），AI 设计「接口功能用例」（人能理解的测试设计层），人审后探索 API 仓库生成待编排用例 → 编排区。

两个入口：**从需求文档**、**从主干链路**（微服务）。与「AI 生成」（会话式、人逐条描述场景）对称互补。

---

## 演进时间线与关键转折

### 阶段 0：起点——为什么要做
**问题**：原有 AI 生成是"人工描述一个场景 → 生成一种场景"，天花板是用户的想象力。用户想要 AI 自主探索、自己判断该测什么。

**最初定位**："AI 自主探索 API 库，生成高质量场景用例——用户给范围，AI 给场景。"

**与 E2E 生成的根本区别**：E2E 是"你说它做"（忠实执行用户描述）；探索是"它想你审"（AI 主动覆盖，人监督）。

---

### 阶段 1：第一次定位反复——"对账/同步"不是定位
**踩的坑**：一开始把功能框成"业务语义 ↔ 用例 的增量同步/对账"，重心放在指纹去重上。

**用户纠偏**：
> "不认，你忽略我的原始需求，我希望 AI 能自主探索，自动生成高质量的场景用例。"

**解法 / 教训**：
- 指纹/同步**降级为隐形的内部机制**（防重复、可累积），不是对外定位。
- **定位 = AI 自主探索生成**，对账只是它"记得上次生成过什么"的副产品。
- 教训：别把"实现机制"当成"产品定位"卖给用户。

---

### 阶段 2：入口之争——不该寄生在 API 仓库
**问题**：最初把入口放在 API 仓库页工具栏（圈接口 → 探索）。
**用户反馈**："ai 探索为什么一定要放在 api 仓库标签，单独开一个 tab 不行吗？"

**解法**：
- 独立 `/ai-explore` 页，不寄生在仓库 tab。
- 撤掉"单接口行内 ✨"——它只省一次勾选，却把交互导向"逐个点"的反模式，与"圈范围"定位相悖。
- 入口收敛为"圈范围"（勾选批量 / 按分类圈选）。

---

### 阶段 3：交互灵魂——"自主"但不"黑盒"
**挑战**：AI 自主了，用户会不会失控、看不懂它要测什么？

**解法（两阶段）**：
1. `explore-plan`：AI 先**亮出场景清单**（打算覆盖哪些场景）给人审，不落库。
2. `explore-generate`：人挑选后才生成用例。

**这是与 E2E 的关键交互差异**：AI 先设计、人监督、再生成。"先出清单再生成"是这个交互的灵魂——让"自主"不黑盒、"高质量"可当场监督。

---

### 阶段 4：技术约束泄漏成痛点——范围大就崩
**问题**：范围一大，AI 一次吃不下 → token 截断 / 上下文过长导致质量稀烂。用户只想"测这个模块"，却被迫为"AI 一次吃不下"买单。

**解法**：后端**自动分批**（≤N 个接口/场景一批，串行跑，进度连续），把"AI 一次吃不下"挡在用户看不见处。**范围可大，质量不塌。**

**配套挑战——网关脏前缀**：公司网关（OpenAI 兼容、实走 Claude）偶尔在 tool arguments 前附垃圾（`{}{"scenarios":[...]}`），直接 `JSON.parse` 在 position 2 报错 → 探索返回空、还看不到日志。
- **解法**：`openai-client.ts` 加 `sanitizeToolArguments`（括号配平扫描提取第一个完整 JSON 对象）；路由侧再加 `parseLooseJsonObject` 兜底。双重容错。

---

### 阶段 5：第二次定位收敛——不是丢 Swagger 让它自由探索
**用户洞察**：
> "并不是直接丢给它一个 swagger 模块文档让它自由探索组织，而是先给他一个接口用例层，让它根据接口用例层描述再去探索 swagger 进行组织。"

**为什么重要**：Swagger 回答"有哪些接口"，但不回答"为什么测、测什么、业务上什么算对"。只丢接口清单，AI 看到了"接口"却不理解"功能"。

**解法**：在"接口"之上加一层**业务意图层**。AI 从业务意图出发，再去 API 仓库探索匹配接口——而不是从接口清单反推。

---

### 阶段 6：第三次定位深化——先生成"人理解的用例"，不是直接执行
**用户洞察**：
> "传入文档，先生成人理解的业务接口功能测试用例，不是直接执行的。"

**关键分层**（三层模型）：

| 层 | 是什么 | 可执行 |
|---|---|---|
| 接口功能用例 | 测试**设计**（人理解：前置/步骤/后置/清理/预期） | 否 |
| 待编排用例 (TestCase) | 可执行**编排草稿** | 是（待确认） |
| 正式用例 | 正式资产 | 是 |

**解法**：新建独立表 `InterfaceFunctionalCase`（不复用 TestCase，避免"是否可执行"语义混淆）。文档 → 功能用例（设计层，可编辑）→ 探索生成 → 待编排。

**教训**：让 AI 先产出"人能审、能改"的测试设计，而不是直接跳到不可控的可执行结构——既符合 AI 能力边界（擅长读文档/拆场景，不擅长精确接口映射），又给了人审环节。

---

### 阶段 7：关联追溯漏洞——两条路径行为不一致
**发现的 bug**：从 `/ai-explore` 不点"保存"直接探索生成时，功能用例只活在内存、从没入库 → 派生的 TestCase 源头丢失、追溯断。而从管理页触发时却能追溯。同一动作两页面能力不一致。

**解法（方案 A）**：探索生成 = 人工评审后沉淀为正式用例。
- **后端统一 upsert**：`explore-generate` 收到功能用例时，没 id 就建、有 id 就标记 `generated`，再生成 TestCase。
- **双向可追溯**：功能用例 `generatedCaseIds`（正向）↔ TestCase `sourceFunctionalCaseId`（反向，新增字段）。
- 与既有「业务语义 → 用例」`sourceFingerprint` 派生关系**同构**，不另造一套。

**教训**："保存"和"生成"两个动作不绑定，就会让追溯能力分叉——收口到后端一处。

---

### 阶段 8：长文档 token 爆仓——分段 + 三道闸
**问题**：塞一篇 1 万字需求文档直接爆仓。

**分析——两个独立方向**：
- **输入爆**：文档太长一次喂不下 → 文档**分段**（按 Markdown 标题优先，否则按段落边界，不切碎句子）。
- **输出爆**：一次让 AI 吐几十条完整用例 → 限制**单段产出条数**。

**三道闸**（超限一律截断 + 明示提示，绝不静默）：
1. 输入文档总长上限（`MAX_DOC_CHARS`）
2. 分段数上限（`MAX_SEGMENTS`）
3. 单段产出条数上限（`MAX_CASES_PER_SEG`）

**配套**：每段带"全局上下文头"（模块名 + 文档开头摘要）避免分段丢背景；串行逐段、单段失败记 `failedSegments` 不静默；合并按 title 去重。

**教训**：技术约束（窗口上限）不能泄漏成用户痛点——后端自动分段，用户只管粘文档。

---

### 阶段 9：微服务的根本挑战——文档与测试正交
**用户提问**：现实中是按模块/服务切的业务文档（微服务），从用户角度要测一连串微服务，怎么整合？

**核心矛盾**：
```
文档侧（按服务切）          测试侧（按用户业务流切）
账户服务文档                用户下单流程：
订单服务文档                  账户 → 订单 → 库存 → 支付 → 通知
库存服务文档                （横跨 5 个服务，但没有任何一篇文档描述它）
```
- 每篇模块文档只讲**服务内**功能，不知道跨服务的上下游。
- 真正值钱的跨服务 E2E（异常/回滚/对账），其知识**不在任何单篇文档里**。

**这动摇了"导入即生成"的前提**：微服务下一篇文档不是一个完整测试范围，只是一块拼图。直接喂模块文档，只能生成"服务内碎片用例"，拼不出跨服务链。

---

### 阶段 10：覆盖率的真相——一句意图覆盖不了多少
**用户追问**：一句业务意图能覆盖多少场景？

**诚实结论**：
```
一句意图        → 锚定主干 happy path        覆盖 ~10-20%
+ 服务文档/语义 → 每个服务的规则/副作用/约束   覆盖 ~50%
+ AI 测试方法论 → 沿链派生异常/对账/边界       覆盖 ~80%
（剩 ~20% 业务特有的，永远要人补）
```

**关键认知**：覆盖率不取决于意图多长，取决于 **AI 沿意图锚定的链，能不能调动"服务能力素材"主动发散异常和对账**。意图定骨架，AI 填血肉，文档/语义供依据。

---

### 阶段 11：职责划分定型（B 方案）
**用户拍板**：
> "测试人员只梳理主干场景（创建订单→风控→资产→冻结购买力→...→推送上手），整个链路的异常和其他分支由 AI 构造。"

**最终职责表**：

| 谁 | 干什么 | 不干什么 |
|---|---|---|
| 测试人员 | 梳理主干骨架 + 关联各节点服务能力 + 最后审阅补漏 | 不逐条想异常/对账 |
| AI | 沿主干发散异常/对账/边界，产出**功能用例**(设计层) | 不猜业务规则(从语义来)、不直接出可执行结构 |
| 服务能力/语义 | 提供"每个节点什么算对"的依据 | —— |
| 探索生成 | 把审定的功能用例匹配接口、组装待编排 | —— |

**两个守住的边界**：
1. AI 发散不能凭空——关联了用例的节点用准确规则，没关联的标注"需人工确认"，不编造预期。
2. AI 产出的是测试设计（功能用例），不直接产可执行 flowConfig（分层别塌）。

**喂法（已定）**：节点文本 + 可选关联已沉淀用例（关联准、不关联 AI 模糊匹配并标 unmatched）。

---

### 阶段 12：收敛动作——砍掉相悖的旧入口
**决策**：去掉"从接口范围"入口（圈一批接口让 AI 反推场景）。
**为什么**：它从接口清单出发，正是阶段 5 否掉的方向，与"从业务意图出发"主线相悖，留着只会让用户困惑用哪个。
**做法**：摘前端入口，保留 `explore-plan`/scenarios 后端代码（降低误伤风险）。

**关于"单文档要不要加选接口范围"**：判断为**之后再做**，且应做在下游——接口范围约束不该在"设计用例"时管，该在"匹配接口"（explore-generate）时管。分层别提前糊。

---

## 贯穿全程的几条原则（沉淀）

1. **定位 ≠ 实现机制**：指纹/同步是机制，不是定位（阶段 1）。
2. **从业务意图出发，不从接口清单出发**：文档/意图是知识入口，接口是知识库，AI 负责连接（阶段 5）。
3. **先出"人能审的设计"，再到"可执行"**：分层清晰，每层有人审环节，AI 不跳层直接产底层结构（阶段 6、11）。
4. **技术约束不泄漏成用户痛点**：分批、分段、容错都挡在后端，用户只管给意图/范围（阶段 4、8）。
5. **失败/截断绝不静默**：一律明示提示（failedCount / 截断条 / unmatched）（阶段 4、8、11）。
6. **追溯收口到一处**：避免多入口行为分叉（阶段 7）。
7. **覆盖率诚实暴露**：不夸大"测全了"，明示"覆盖已知维度，业务特有仍需人补"（阶段 10）。
8. **人给骨架/意图，AI 填血肉，素材供依据**：三者各司其职（阶段 11）。
